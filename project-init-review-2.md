project-init-review.md의 "CLAUDE.md “설계 원칙” 섹션 추가", "CLAUDE.md “기능-테이블-API 매핑” 섹션 추가" 두 항목을 각각 추가하는 방안과 추가하지 않는 방안을 검토했으며 추가하지 않는 항목은 대체 방안을 검토했다.

## 의견: 선택적 추가 권장

두 섹션의 특성이 다르므로 각각 다르게 접근하는 것이 좋습니다.

---

### 설계 원칙 — **추가 권장**

**이유:**

| 측면 | 분석 |
|------|------|
| **분량** | 5줄 (약 100자) — CLAUDE.md 길이 증가 미미 |
| **참조 빈도** | 개발 의사결정 시점마다 (중간 빈도) |
| **없을 때 문제** | "왜 오프라인 우선으로 설계했지?" → PRD 참조 필요 |
| **대체 가능성** | 낮음 — 다른 섹션에서 커버되지 않는 정보 |

```markdown
## 설계 원칙
1. **오프라인 우선**: 감정 기록은 WatermelonDB에 즉시 저장
2. **최소 외부 의존**: MVP에서 AI/ML 불필요, SQL 집계로 분석
3. **점진적 복잡성**: 규칙 기반 → 통계 기반 → LLM 기반
4. **RLS 기반 보안**: 모든 테이블에 Row Level Security
5. **낙관적 UI**: 저장/수정 시 로컬 상태 즉시 반영
```

5줄로 개발 의사결정의 **근거**를 제공합니다. 비용 대비 효용이 높습니다.

---

### 기능-테이블-API 매핑 — **추가하지 않음 권장**

**이유:**

| 측면 | 분석 |
|------|------|
| **분량** | 9행 테이블 + 비고 (약 500자) — 상당한 증가 |
| **중복성** | DB 규칙(테이블 목록) + API 규칙(엔드포인트 목록)과 **정보 중복** |
| **참조 빈도** | 기능 구현 시 (높지만, 기능당 1회) |
| **대체 가능성** | 높음 — PRD 참조 한 줄로 대체 가능 |

**대안**: PRD 참조 섹션에 한 줄 추가

```markdown
## PRD 참조
- 기능 구현 시 테이블/API 매핑: PRD Section 7-8
- 상세 DB 스키마 (CREATE TABLE): PRD Section 7-5
- 화면 UI 명세: PRD Section 8
- 수용 기준(AC): PRD Section 5
```

이미 CLAUDE.md에 **DB 규칙**(테이블 목록, RLS)과 **API 규칙**(엔드포인트 목록)이 있으므로, 매핑표까지 추가하면 **같은 정보가 3곳에 분산**됩니다.

### 결론

| 섹션 | 결정 | 근거 |
|------|------|------|
| 설계 원칙 | **추가** | 짧고, 다른 섹션에서 커버 안 됨, 의사결정 기준 제공 |
| 기능-테이블-API 매핑 | **추가 안 함** | 길고, DB/API 규칙과 중복, PRD 참조로 대체 |

이 방식이면:
- CLAUDE.md 간결성 유지 (5줄만 증가)
- 핵심 의사결정 기준은 즉시 참조 가능
- 상세 매핑은 필요 시 PRD에서 참조
